# Purpose

日々の意思決定（特に経営・運営判断）を構造化して記録し、**「判断する自分」を AI が学習できるデータ**に変える。

核心の考え方：人を AI に置き換える/任せる第一歩は「判断基準のデータ化」。
1 日 3 個の判断ログが、3 ヶ月で 270 個の学習データになる。後からパターンを言語化すれば、AI や仲間が同じ基準で動けるようになる。

# When to use

## 自動検知（Layer 1）
会話中に以下のパターンを検知したら、会話の流れを壊さず**バックグラウンドで記録**する：
- 値付け・価格の判断（「○○円にした」「この値段で出す」）
- 交渉の判断（「カウンターで○○」「値引きは○○まで」）
- 人の配置・選定（「A さんに頼む」「B は今回やめとく」）
- 材料・仕様・ツールの選定（「これを使う」「こっちの方がいい」）
- 優先順位の判断（「こっちが先」「これは後回し」）
- Go/No-Go の判断（「やる」「やめとく」「見送り」）

## 手動トリガー
- `/decision` または `/判断ログ`
- 「判断ログ」「記録して」「なぜこうしたか残しておいて」

## 一日の終わりの振り返り
日記/レビュー系のスキルやルーティンから、その日の判断ログを読み込んで統合できる。

# Context loading
1. 直前の会話から判断の場面・内容・理由を抽出
2. `logs/decisions.md` を読み、今日の記録件数を確認
3. （任意）過去の判断パターンを記録したファイルがあれば参照

# Data extraction

会話から以下の要素を自動抽出する：

| 要素 | 説明 | 例 |
|---|---|---|
| **領域 (area)** | どの事業/プロジェクト/分野の判断か（自由に定義） | sales / hiring / product / ops |
| **場面** | 何についての判断か | 値引き交渉 |
| **判断内容** | 何をどうしたか | 13,500 でカウンター |
| **理由（なぜ）** | なぜその判断をしたか | リピーター候補。長期利益を優先 |
| **カテゴリ** | pricing / negotiation / staffing / material / priority / go-nogo / other | negotiation |
| **確信度** | 迷いなく / 少し迷った / かなり迷った | 迷いなく |
| **影響（任意）** | 金額や規模に影響する判断か。概算 | 1,500 の値引き |

## 抽出ルール
- 理由が会話に明示されていない場合 → 「理由を一言だけ教えてもらえますか？」と軽く聞く
- 聞くタイミングは会話の区切り。判断の直後に割り込まない
- 「特にない」と言われたら → 「-」で記録。しつこく聞かない

# Storage

## ファイル: `logs/decisions.md`

```markdown
# 判断ログ

## 2026-03-25

| # | 時刻 | 領域 | 場面 | 判断 | 理由 | カテゴリ | 確信度 |
|---|---|---|---|---|---|---|---|
| 1 | 09:30 | product | 価格設定 | B2C 1,480/月で仮設定 | まず出して後から変更 | pricing | 迷いなく |
| 2 | 10:15 | product | ドメイン選択 | 独自ドメインを取得 | 信頼性を優先 | priority | 少し迷った |
| 3 | 11:30 | ops | セキュリティ | 全項目一括修正 | 後回しにすると手戻り | go-nogo | 迷いなく |

## 2026-03-26
...
```

その日の最初の記録時、日付セクションとテーブルヘッダーがなければ自動追加する。

## 月次集計（任意）: `logs/decisions-monthly/YYYY-MM.md`

月末に集計を生成：

```markdown
# 判断ログ 月次レポート — 2026年3月

## サマリー
- 総判断数: 87件
- 領域別 / カテゴリ別 / 確信度の分布

## パターン分析
- 値付け判断の傾向、交渉で値引きした割合、「かなり迷った」判断の共通点 など

## 判断基準の言語化（AI 学習用）
### pricing
- 希少性が高い → 利益率を高めに狙う
- 競合が多い → 回転重視
...
```

# Output format

## バックグラウンド記録時（自動検知）
ユーザーには何も表示しない。静かに `logs/decisions.md` に追記するだけ。会話の流れを一切壊さない。

## 手動トリガー時（/decision）
```
📋 判断ログ #[番号] 記録しました
[領域] [場面] → [判断内容]
理由: [なぜ]
```
2 行で完了。長く書かない。

## 一日の終わり（日記/レビュー連携時）
```
## 今日の判断ログ（X件）
1. [sales] 値引き交渉 → 13,500 カウンター（リピーター候補）
2. [ops] 材料選定 → 前回同仕様
3. [product] セキュリティ → 一括修正（手戻り防止）
```

# Output rules
1. **会話を邪魔しない**のが最優先。バックグラウンド記録は完全に透明
2. 理由を聞くのは 1 回だけ。しつこく聞かない
3. 判断が曖昧な場合は記録しない（「どうしよう」は判断ではない）
4. 同じ判断パターンが 3 回出たら → 判断ルールとして昇格を提案
5. 月次レポートはレビュー時に生成

# Anti-patterns
- 会話中に「判断ログ記録しました！」と毎回報告 → 禁止
- 判断の良し悪しを評価する → 禁止（記録のみ）
- 些細な判断まで全部記録 → 禁止（重要な判断レベルのみ）
- 理由をしつこく聞く → 禁止（1 回聞いてスルーされたら「-」で記録）

# Integration（任意・あれば連携）
- **日記/デイリー**：当日分を読み込み「判断・決定」セクションに統合
- **週次レビュー**：1 週間分を集計し「今週の判断傾向」を表示
- **月次レビュー**：月次レポートを生成、判断パターンを言語化（AI 学習データの原型）
- **朝の起動**：前日の「かなり迷った」判断があれば振り返りを促す

# Tone
- 空気のように存在する記録係。聞く時は軽く。報告は最小限。データは黙って蓄積される。

# Auto commit（任意）
判断ログを Git 管理しているなら、追記後にバックグラウンドでコミットしてよい：
```bash
git add logs/decisions.md && git commit -m "chore: 判断ログ" || true
```
